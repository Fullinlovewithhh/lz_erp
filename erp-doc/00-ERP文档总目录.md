> 整理日期：2026-07-19

> 使用原则：先确认业务，再设计数据，最后实现代码。

## 1. 文档权威顺序

发生内容冲突时，按以下顺序判断：

1. `01-业务真源` 中已经确认的业务规则。
2. `02-数据设计` 中与业务规则对应的数据结构。
3. `03-技术与实施` 中的实现、兼容和迁移说明。
4. README、代码注释和 Git 历史只用于辅助核对。

业务规则变更时，应先修改业务真源文档，再修改数据库、后端和前端。代码已经实现但业务文档没有确认的内容，不能自动视为正式业务规则。

## 3. 当前确认范围

已经形成正式文档：

- 客户、客户订单和客户原始 CAD。
- CAD 评审池、拆单草稿和工厂订单生成。
- L 实木生产线 ERP 报价。
- V 板式生产线外部报价登记。
- 产品自动计价、特殊加价、五金和折扣。
- 客户订单报价汇总、PDF、客户确认和进入财务的条件。
- 全款、分期、月结和免单付款方案。
- 付款计划、到账确认、厂长审批和客服下料放行。
- 月结客户逾期锁定和付款提醒。
- 补单的建立、报价、财务处理和独立下料放行。

仍需逐段确认并新增文档：

- 下料、BOM、板材、五金锁库和采购申请。
- 排产、生产工序、质检、包装和二维码追踪。
- 补单下料后的 BOM、锁库、生产与追踪流程。
- 售后责任判断与免费重做审批细节。

未确认部分不得混写进已确认业务流程；先放在新文档的“待确认问题”中。

## 4. 单条业务文档模板

后续每条业务线统一按以下结构编写：

1. 业务目的与范围。
2. 前置条件。
3. 正常业务流程。
4. 参与角色与职责。
5. 业务单据。
6. 状态流转。
7. 字段和必填规则。
8. 修改、撤回、作废和异常规则。
9. 页面与操作权限。
10. 消息提醒。
11. 统计需求。
12. 数据归属。
13. 待确认问题。

流程文档只描述业务事实，不记录“后续要改哪些代码”。开发任务应进入 Git 提交、Issue 或单独的实施清单。

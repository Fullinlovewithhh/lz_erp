# 数据库主结构说明

本文档用于固定 ERP 系统的数据库主结构。后续修改订单、报价、财务、排产、生产等模块时，优先遵循本文档的主数据关系，避免字段反复堆积。

## 1. 核心主线

当前系统应围绕下面这条主线落地：

```text
客户 customer
  -> 项目 project
      -> 客户订单 customer_order
          -> 工厂订单 contract
              -> 深化分配 quote_assignment
              -> 报价主表 quotation
                  -> 报价明细 quote_detail
                      -> 报价附加费用 quote_detail_extra_price
              -> 收款记录 payment_record
              -> 生产任务 production_task
              -> CAD 文件 contract_cad_file
              -> 财务凭证 finance_voucher
```

业务上必须明确三层订单结构：

```text
项目名称
  -> 客户订单号
      -> 工厂订单号
```

其中：

- 项目名称来自 `project.project_name`。
- 客户订单号来自 `customer_order.customer_order_no`。
- 工厂订单号来自 `contract.factory_order_no`。

## 1.1 编号命名规则

数据库字段命名必须区分内部 ID 和业务编号：

```text
id / xxx_id = 数据库内部主键或关系字段，系统使用
xxx_no      = 业务单号或编号，用户可见、可搜索、可打印
xxx_code    = 主数据编码，用户可见、可搜索
xxx_name    = 名称
xxx_snapshot = 历史快照字段
```

当前主链路固定为：

```text
customer.id                         = 客户ID
customer.customer_code              = 客户编码

project.project_id                  = 项目ID
project.project_no                  = 项目编号

customer_order.id                   = 客户订单ID
customer_order.customer_order_no    = 客户订单号

contract.id                         = 工厂订单ID
contract.factory_order_no           = 工厂订单号
```

规则：

- 不把“ID”和“号”混用。
- 用户输入和搜索业务编号，例如客户订单号、工厂订单号。
- 系统内部绑定关系使用 ID，例如 `customer_order_id`、`contract.id`。
- `contract_no` 是历史兼容字段，不再作为新的业务主字段。

## 2. 主表职责

### 2.1 customer 客户表

`customer` 是客户主数据表，用来维护客户档案。

建议主字段：

| 字段 | 含义 | 说明 |
| --- | --- | --- |
| `id` | 客户 ID | 系统主键 |
| `customer_code` | 客户编号 | 应唯一 |
| `customer_name` | 客户名称 | 当前客户档案名称 |
| `phone` | 联系电话 | 当前客户档案电话 |
| `address` | 客户地址 | 当前客户档案地址 |
| `level` | 客户等级 | 可用于客户分层 |
| `owner` | 客户负责人 | 通常是客服或业务负责人 |
| `custom_fields` | 扩展字段 | 试运行可以用，正式落地后常用字段应拆出 |
| `created_at` / `updated_at` | 创建和更新时间 | 审计基础字段 |

落地规则：

- 客户档案可以被订单快照引用。
- 已有关联项目或订单的客户，不建议物理删除。
- 常用经营字段不要长期藏在 `custom_fields` 中。

### 2.2 project 项目表

`project` 是客户下面的项目层。

建议主字段：

| 字段 | 含义 | 说明 |
| --- | --- | --- |
| `project_id` | 项目 ID | 系统主键 |
| `project_no` | 项目编号 | 应唯一 |
| `project_name` | 项目名称 | 第一层业务名称 |
| `customer_id` | 所属客户 | 关联 `customer.id` |
| `project_address` | 项目地址 | 项目现场或交付地址 |
| `project_status` | 项目状态 | 如进行中、已结束、停用 |
| `project_manager` | 项目负责人 | 可选 |
| `remark` | 备注 | 可选 |
| `is_deleted` | 是否删除 | 建议软删除 |
| `created_at` / `updated_at` | 创建和更新时间 | 审计基础字段 |

落地规则：

- 一个客户可以有多个项目。
- 一个项目可以有多个客户订单号。
- 项目已有关联客户订单或工厂订单后，不建议物理删除。

### 2.3 customer_order 客户订单表

`customer_order` 是当前新增的关键中间层，用来承接客户提供的订单号。

建议主字段：

| 字段 | 含义 | 说明 |
| --- | --- | --- |
| `id` | 客户订单 ID | 系统主键 |
| `customer_id` | 所属客户 | 关联 `customer.id` |
| `project_id` | 所属项目 | 关联 `project.project_id` |
| `customer_order_no` | 客户订单号 | 客户提供的订单号 |
| `customer_order_name` | 客户订单名称 | 可选，用于内部识别 |
| `remark` | 备注 | 可选 |
| `status` | 状态 | 如启用、停用、作废 |
| `is_deleted` | 是否删除 | 建议软删除 |
| `created_at` / `updated_at` | 创建和更新时间 | 审计基础字段 |

落地规则：

- 一个项目可以有多个客户订单号。
- 一个客户订单号下面可以有多个工厂订单号。
- 建议唯一约束：`project_id + customer_order_no`。
- 客户订单号是客户侧订单维度，不等于工厂订单号。

### 2.4 contract 工厂订单表

`contract` 当前承担工厂订单主表职责。业务命名上应统一理解为“工厂订单”，不是传统合同表。

建议主字段：

| 字段 | 含义 | 说明 |
| --- | --- | --- |
| `id` | 工厂订单 ID | 系统主键 |
| `factory_order_no` | 工厂订单号 | 工厂内部唯一订单号 |
| `customer_id` | 所属客户 | 冗余关系字段，便于查询 |
| `project_id` | 所属项目 | 冗余关系字段，便于查询 |
| `customer_order_id` | 所属客户订单 | 正式主关系字段 |
| `status` | 工厂订单状态 | 主流程状态 |
| `production_progress` | 生产进度 | 生产阶段使用 |
| `demand_desc` | 需求说明 | 订单需求 |
| `created_at` / `updated_at` | 创建和更新时间 | 审计基础字段 |

建议快照字段：

| 字段 | 含义 | 说明 |
| --- | --- | --- |
| `customer_name_snapshot` | 客户名称快照 | 下单时客户名称 |
| `customer_phone_snapshot` | 客户电话快照 | 下单时客户电话 |
| `customer_address_snapshot` | 客户地址快照 | 下单时客户地址 |
| `project_name_snapshot` | 项目名称快照 | 下单时项目名称 |
| `customer_order_no` | 客户订单号快照 | 当前保留兼容，后续可改名为 `customer_order_no_snapshot` |

历史兼容字段：

| 字段 | 建议处理 |
| --- | --- |
| `contract_no` | 当前与工厂订单号同值，后续不再作为业务主字段 |
| `customer_name` | 建议逐步并入快照字段 |
| `customer_phone` | 建议逐步并入快照字段 |
| `customer_address` | 建议逐步并入快照字段 |

落地规则：

- 工厂订单必须属于一个客户订单。
- 工厂订单号以 `factory_order_no` 为准。
- `contract_no` 只作为历史兼容字段，不应继续扩展新逻辑。
- 订单显示时应按 `project_name -> customer_order_no -> factory_order_no` 展示。

## 3. 业务子表关系

### 3.1 quote_assignment 深化分配表

用途：表示工厂订单进入深化订单池、被客服指定工程师，或被深化工程师领取。

主关系：

```text
quote_assignment.contract_id -> contract.id
```

字段建议：

| 字段 | 含义 |
| --- | --- |
| `contract_id` | 所属工厂订单 |
| `service_staff` | 客服 |
| `engineer` | 深化工程师 |
| `status` | 分配状态，如 PENDING_CLAIM、ASSIGNED、QUOTED |
| `remark` | 备注 |

注意：

- `contract_no` 如果保留，只能作为快照或兼容字段。
- 分配关系应以 `contract_id` 为准。

### 3.2 quotation 报价主表

用途：报价单主记录，记录报价总金额、报价人、报价说明。

主关系：

```text
quotation.contract_id -> contract.id
quotation.assignment_id -> quote_assignment.id
```

落地规则：

- 一个工厂订单可以有多个报价单。
- 一个深化分配可以产生一个或多个报价版本。
- 后续如果要做正式版本管理，建议新增 `quotation_version` 或在 `quotation` 上补版本字段。

### 3.3 quote_detail 报价明细表

用途：报价单中的产品、材质、尺寸、数量、面积、单价、金额。

主关系：

```text
quote_detail.quote_id -> quotation.id
```

落地规则：

- 报价明细属于报价主表，不直接属于工厂订单。
- 所有报价金额汇总应从明细计算到主表。

### 3.4 quote_detail_extra_price 报价附加费用表

用途：记录每条报价明细应用的加价规则和附加费用。

主关系：

```text
quote_detail_extra_price.quote_id -> quotation.id
quote_detail_extra_price.detail_id -> quote_detail.id
```

落地规则：

- 附加费用应绑定具体报价明细。
- 如果是整单加价，后续建议单独增加整单费用表，不要硬塞进明细。

### 3.5 payment_record 收款记录表

用途：记录财务确认、收款金额、收款渠道、审核人。

主关系：

```text
payment_record.contract_id -> contract.id
```

落地规则：

- 收款记录属于工厂订单。
- 允许多次收款。
- 不建议物理删除收款记录，真实落地应改为作废或冲销。

### 3.6 production_task 生产任务表

用途：记录工厂订单进入生产后的任务。

主关系：

```text
production_task.contract_id -> contract.id
```

落地规则：

- 生产任务必须来自已满足财务条件的工厂订单。
- 后续拆单、BOM、板材清单、五金清单应挂在生产任务或工厂订单下面，具体取决于拆单粒度。

### 3.7 contract_cad_file CAD 文件表

用途：记录工厂订单相关 CAD 文件。

主关系：

```text
contract_cad_file.contract_id -> contract.id
```

注意：

- `contract_no` 只作为文件上传时的快照显示。
- 文件归属以 `contract_id` 为准。

### 3.8 finance_voucher 财务凭证表

用途：记录财务上传的凭证文件。

主关系：

```text
finance_voucher.contract_id -> contract.id
```

注意：

- `contract_no` 只作为凭证上传时的快照显示。
- 凭证归属以 `contract_id` 为准。

## 4. 建议外键关系

正式落地前建议补齐以下外键或等价的数据库约束：

```text
project.customer_id -> customer.id

customer_order.customer_id -> customer.id
customer_order.project_id -> project.project_id

contract.customer_id -> customer.id
contract.project_id -> project.project_id
contract.customer_order_id -> customer_order.id

quote_assignment.contract_id -> contract.id

quotation.contract_id -> contract.id
quotation.assignment_id -> quote_assignment.id

quote_detail.quote_id -> quotation.id

quote_detail_extra_price.quote_id -> quotation.id
quote_detail_extra_price.detail_id -> quote_detail.id

payment_record.contract_id -> contract.id
production_task.contract_id -> contract.id
contract_cad_file.contract_id -> contract.id
finance_voucher.contract_id -> contract.id
```

如果短期担心外键影响试运行，可以先补服务层校验和索引；正式上线前再补数据库外键。

## 5. 建议唯一约束

建议至少保留：

```text
customer.customer_code UNIQUE
project.project_no UNIQUE
customer_order(project_id, customer_order_no) UNIQUE
contract.factory_order_no UNIQUE
production_task.task_no UNIQUE
staff.staff_code UNIQUE
product.product_code UNIQUE
material.material_code UNIQUE
```

注意：

- `contract_no` 当前已有唯一约束，但未来不应继续作为主要业务编号。
- 如果 `contract_no` 与 `factory_order_no` 长期同值，可以先两者保持唯一；后续稳定后再决定是否移除 `contract_no`。

## 6. 快照字段规则

订单、报价、财务、生产这些业务单据需要保留历史现场，因此允许快照字段。

正确用法：

```text
客户当前名称：customer.customer_name
订单下单时客户名称：contract.customer_name_snapshot
```

规则：

- 主关系用 ID 字段。
- 历史显示用快照字段。
- 查询统计优先用 ID 关联。
- 快照字段不参与主关系判断。
- 快照字段只在创建单据或关键变更时写入。

### 7.3 custom_fields / custom_text 字段

这些字段适合试运行阶段快速扩展，但不能无限使用。

建议：

- 临时、不常查询的字段可以放 `custom_fields`。
- 经常搜索、统计、权限判断、导入导出的字段，应拆成正式字段。
- 客户结算方式、账期、报价体系、内部折扣等经营字段，后续应从 JSON 中拆出。

## 8. 状态字段规则

工厂订单主状态建议统一为：

```text
报价中
待客户确认
待财务确认
财务已收款
待排产
生产中
已完成
待发货
已发货
```

报价分配状态建议作为子状态：

```text
PENDING_CLAIM  待领取
ASSIGNED       已分配
QUOTED         已报价
```

规则：

- 工厂订单主状态只表达大流程。
- 报价池、工程师领取、报价明细提交属于报价子流程。
- 财务收款记录属于财务子流程。
- 生产任务进度属于生产子流程。

## 9. 删除规则

正式落地建议：

- 主业务数据不物理删除。
- 客户、项目、客户订单、工厂订单应使用软删除或作废。
- 报价、收款、生产记录原则上不删除，只允许作废、冲销或变更。
- 所有关键删除、作废、变更都应写入 `operation_log`。

## 10. 下一步落地顺序

建议按下面顺序整理：

1. 更新 `init.sql`，把 `customer_order` 和 `contract.customer_order_id` 写成正式结构。
2. 增加必要索引和唯一约束。
3. 明确 `contract.factory_order_no` 为唯一业务编号。
4. 把 `contract_no`、`customer_order_no`、`contract_no` 冗余字段标记为兼容或快照。
5. 补外键或至少补等价服务层校验。
6. 整理 `custom_fields` 中已经稳定的正式字段。
7. 把启动自动改表逻辑逐步替换为明确的迁移脚本。

## 12. 总结

当前数据库不是推倒重来的问题，而是需要把已经形成的业务逻辑正式固化：

```text
客户 -> 项目 -> 客户订单 -> 工厂订单 -> 报价/财务/生产
```

后续所有新增功能都应围绕这条主线设计字段，避免继续把不同层级的信息堆进同一张表。

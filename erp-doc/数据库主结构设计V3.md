# ERP 数据库主结构设计 V3

## 1. 设计目标

本版本固定 ERP 核心订单结构，解决客户订单和工厂拆单混用的问题。

核心业务链路：

```text
客户 customer
  -> 项目 project（可选）
    -> 客户订单 customer_order
      -> 工厂订单 factory_order（一张或多张）
        -> 报价、深化、排产、生产、财务
```

基本规则：

- 数据表之间使用主键建立关系，不使用名称建立关系。
- 客户和客户订单使用数据库内部数字 ID。
- 工厂订单使用英文数字订单号作为主键，不再重复设置数字 ID 和工厂订单号。
- 前端不显示客户 ID、客户订单 ID 等内部数字 ID。
- 一张客户订单可以拆成一张或多张工厂订单。
- 一张工厂订单只能属于一张客户订单。
- 客户原始 CAD 属于客户订单，深化图和生产图属于工厂订单。

## 2. 前后端字段口径

| 业务对象 | 后端主键 | 前端主要显示 | 说明 |
|---|---|---|---|
| 客户 | `customer.id` | `customer_name` 客户名称 | 数字 ID 只在后端使用 |
| 项目 | `project.project_id` | `project_name` 项目名称 | 项目可以不填写 |
| 客户订单 | `customer_order.id` | `customer_order_no`、`customer_order_name` | 客户提供的订单 |
| 工厂订单 | `factory_order.factory_order_id` | `factory_order_id`、`factory_order_name` | `factory_order_id` 同时是主键和工厂订单号 |

字段命名规则：

```text
id / xxx_id       主键或关联键
xxx_no            普通业务编号
xxx_code          基础资料编码
xxx_name          名称
xxx_snapshot      历史快照
```

特殊约定：`factory_order_id` 是英文数字组成的业务主键，中文统一显示为“工厂订单号”。

## 3. 客户表 customer

```text
customer
  id                 BIGINT PRIMARY KEY AUTO_INCREMENT
  customer_code      VARCHAR(50) NOT NULL
  customer_name      VARCHAR(100) NOT NULL
  phone              VARCHAR(30)
  address            VARCHAR(255)
  status             VARCHAR(30) NOT NULL
  created_at         DATETIME NOT NULL
  updated_at         DATETIME NOT NULL
```

约束：

- `customer_code` 全系统唯一。
- 客户名称用于前端选择和模糊搜索，不作为表之间的关联字段。
- 客户已有业务数据后只能停用，不允许物理删除。

## 4. 项目表 project

```text
project
  project_id         BIGINT PRIMARY KEY AUTO_INCREMENT
  customer_id        BIGINT NOT NULL
  project_no         VARCHAR(50) NOT NULL
  project_name       VARCHAR(100) NOT NULL
  project_address    VARCHAR(255)
  project_status     VARCHAR(30) NOT NULL
  created_at         DATETIME NOT NULL
  updated_at         DATETIME NOT NULL
```

关系：

```text
project.customer_id -> customer.id
```

项目是辅助管理层，不改变客户订单与工厂订单的拆分关系。没有具体项目时，客户订单的 `project_id` 可以为空。

## 5. 客户订单表 customer_order

客户订单表示客户传给工厂的一整张订单，是工厂拆单前的原始业务单元。

```text
customer_order
  id                    BIGINT PRIMARY KEY AUTO_INCREMENT
  customer_id           BIGINT NOT NULL
  project_id            BIGINT NULL
  customer_order_no     VARCHAR(255) NOT NULL
  customer_order_name   VARCHAR(255) NULL
  received_date         DATE NULL
  status                VARCHAR(30) NOT NULL
  remark                TEXT NULL
  is_deleted            TINYINT NOT NULL DEFAULT 0
  created_at            DATETIME NOT NULL
  updated_at            DATETIME NOT NULL
```

关系：

```text
customer_order.customer_id -> customer.id
customer_order.project_id  -> project.project_id
```

约束：

```text
UNIQUE (customer_id, customer_order_no)
```

同一客户下的客户订单号不能重复。不同客户可以使用相同的客户订单号。

建议状态：

```text
待拆单 -> 拆单中 -> 已拆完 -> 已关闭
```

## 6. 工厂订单表 factory_order

工厂订单表示客户订单拆分后，工厂内部实际执行的订单。

```text
factory_order
  factory_order_id       VARCHAR(50) PRIMARY KEY
  customer_order_id      BIGINT NOT NULL
  factory_order_name     VARCHAR(255) NOT NULL
  split_sequence         INT NOT NULL
  engineer_id            BIGINT NULL
  status                 VARCHAR(50) NOT NULL
  demand_desc            TEXT NULL
  created_by             BIGINT NULL
  created_at             DATETIME NOT NULL
  updated_at             DATETIME NOT NULL
```

关系：

```text
factory_order.customer_order_id -> customer_order.id
```

规则：

- `factory_order_id` 同时是数据库主键和前端显示的工厂订单号。
- `factory_order_id` 只能包含英文字母和数字，建议统一大写。
- `factory_order_id` 创建后永久禁止修改。
- 不再设置 `id`、`factory_order_no`、`contract_no` 三套重复编号。
- `factory_order_name` 是拆单名称，例如“住宅1-客厅”。
- `split_sequence` 表示同一客户订单下的拆单顺序。
- 一个工厂订单只能绑定一个 `customer_order_id`。

建议检查约束：

```text
factory_order_id REGEXP '^[A-Z0-9]+$'
split_sequence > 0
UNIQUE (customer_order_id, split_sequence)
```

## 7. 业务示例

客户“客户1”传来住宅1订单：

```text
customer
  id = 1
  customer_name = 客户1

customer_order
  id = 101
  customer_id = 1
  customer_order_no = KH20260001
  customer_order_name = 住宅1

factory_order
  factory_order_id = GC2026000101
  customer_order_id = 101
  factory_order_name = 住宅1-客厅
  split_sequence = 1

factory_order
  factory_order_id = GC2026000102
  customer_order_id = 101
  factory_order_name = 住宅1-卧室
  split_sequence = 2
```

前端显示：

```text
客户1
└── KH20260001 / 住宅1
    ├── GC2026000101 / 住宅1-客厅
    └── GC2026000102 / 住宅1-卧室
```

## 8. 文件归属

### 8.1 客户原始 CAD

```text
customer_order_cad_file
  id                    BIGINT PRIMARY KEY AUTO_INCREMENT
  customer_order_id     BIGINT NOT NULL
  file_name             VARCHAR(255) NOT NULL
  file_ext              VARCHAR(20)
  file_size             BIGINT
  file_path             VARCHAR(500) NOT NULL
  created_by            BIGINT NULL
  created_at            DATETIME NOT NULL
```

关系：

```text
customer_order_cad_file.customer_order_id -> customer_order.id
```

客户原始 CAD 属于客户订单，其下所有工厂订单都可以查看。

### 8.2 深化和生产文件

```text
factory_order_design_file
  id                    BIGINT PRIMARY KEY AUTO_INCREMENT
  factory_order_id      VARCHAR(50) NOT NULL
  file_type             VARCHAR(30) NOT NULL
  file_name             VARCHAR(255) NOT NULL
  file_ext              VARCHAR(20)
  file_size             BIGINT
  file_path             VARCHAR(500) NOT NULL
  created_by            BIGINT NULL
  created_at            DATETIME NOT NULL
```

关系：

```text
factory_order_design_file.factory_order_id -> factory_order.factory_order_id
```

## 9. 下游业务关联

报价、深化分配、流程日志、排产、生产、付款和财务凭证都应使用：

```text
factory_order_id VARCHAR(50)
```

主要关系：

```text
quote_assignment.factory_order_id -> factory_order.factory_order_id
quotation.factory_order_id        -> factory_order.factory_order_id
workflow_log.factory_order_id      -> factory_order.factory_order_id
production_task.factory_order_id   -> factory_order.factory_order_id
payment_record.factory_order_id    -> factory_order.factory_order_id
finance_voucher.factory_order_id   -> factory_order.factory_order_id
```

下游表可以保存工厂订单名称、客户订单号或客户名称快照用于历史展示，但快照不能代替正式关联字段。

## 10. 前端操作逻辑

新建客户订单：

1. 用户选择客户名称，前端保存对应的 `customer.id`。
2. 用户输入客户订单号和客户订单名称。
3. 后端自动创建 `customer_order.id` 并建立客户关系。
4. 内部 ID 不在前端显示。

拆分工厂订单：

1. 用户按客户名称或客户订单号搜索客户订单。
2. 选择客户订单后，新建一张或多张工厂订单。
3. 系统生成或录入唯一的 `factory_order_id`。
4. 用户填写 `factory_order_name`，例如“住宅1-客厅”。
5. 后端保存 `customer_order_id`，形成严格的一对多关系。

## 11. 删除和修改规则

- 客户、项目和客户订单使用停用或软删除。
- 已产生工厂订单的客户订单不能物理删除。
- 已被下游表引用的工厂订单不能物理删除。
- `factory_order_id` 创建后不可修改；录错时应作废并重新创建。
- 客户订单号如需修改，应记录操作日志。
- 禁止通过客户名称、客户订单号或工厂订单名称直接关联业务表。

## 12. 现有数据库迁移方案

当前系统的 `contract` 表实际承担工厂订单职责。迁移分阶段执行。

第一阶段：补齐语义。

- 给现有 `contract` 增加 `factory_order_name`。
- 确保每条数据都有有效 `customer_order_id`。
- 停止新增对 `contract_no` 的业务依赖。
- 将客户原始 CAD 迁移到 `customer_order_cad_file`。

第二阶段：建立新表并迁移。

- 创建正式的 `factory_order` 表。
- 使用现有 `factory_order_no` 生成 `factory_order.factory_order_id`。
- 将现有 `contract.id` 与新 `factory_order_id` 的对应关系保存到迁移映射表。
- 逐表迁移报价、深化、流程、排产、生产和财务关联字段。

第三阶段：切换应用。

- 后端统一使用 `factory_order_id`。
- 前端中文统一显示“工厂订单号”和“工厂订单名称”。
- 完成数据一致性检查后，将 `contract`、`contract_no` 和 `factory_order_no` 转为历史兼容结构。
- 稳定运行并完成备份后，再决定是否删除旧字段和旧表。

## 13. 最终权威关系

```text
customer.id
  -> project.customer_id
  -> customer_order.customer_id

project.project_id
  -> customer_order.project_id（可空）

customer_order.id
  -> factory_order.customer_order_id
  -> customer_order_cad_file.customer_order_id

factory_order.factory_order_id
  -> factory_order_design_file.factory_order_id
  -> quotation.factory_order_id
  -> quote_assignment.factory_order_id
  -> workflow_log.factory_order_id
  -> production_task.factory_order_id
  -> payment_record.factory_order_id
  -> finance_voucher.factory_order_id
```

本结构作为下一轮 `init.sql`、Java 实体、接口和前端字段重构的目标结构。目标结构未完成迁移前，旧 `contract` 表只作为兼容层保留。

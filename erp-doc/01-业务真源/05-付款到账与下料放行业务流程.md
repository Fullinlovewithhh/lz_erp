# 付款、到账与下料放行业务流程

> 文档性质：业务真源
>
> 状态：付款计划、到账确认、厂长审批和客服下料放行已经确认
>
> 当前边界：流程结束于“客服已放行、等待下料”，不定义 BOM、锁库和实际生产操作

## 1. 文档范围

本文档定义客户报价确认后至客服取得并执行下料放行权之间的流程，包括：

- 付款方案选择。
- 付款计划建立、修改和审批。
- 客户付款通知与财务到账确认。
- 实际到账对付款计划的冲抵。
- 厂长下料放行审批。
- 客服执行下料放行。
- 月结逾期控制、提醒、撤回和操作留痕。

本文档不定义下料明细、BOM、板材清单、五金锁库、采购、排产和生产工序。

## 2. 前置条件

客户订单进入本流程前必须满足：

- 所有 L 工厂订单报价已经完成，当前 L 报价 PDF 已经客户确认。
- 所有 V 工厂订单已经登记当前有效外部报价并分别完成客户确认。
- 折扣修改已经完成留痕，成交价调整已经完成厂长审批。
- 系统已经形成客户订单当前最终应收。
- 当前报价、PDF和客户确认不存在失效或正在修改的情况。

L 与 V 分别报价、分别确认，但付款计划和财务应收统一绑定客户订单。

## 3. 付款方案

客户订单进入财务前，由客服选择一种付款方案。

| 付款方案 | 定义 | 付款计划审批 | 下料放行审批 |
|---|---|---|---|
| 全款 | 客户计划一次付清当前最终应收 | 不需要 | 财务确认全额到账后不需要 |
| 分期付款 | 客户按多期金额和日期付款 | 厂长必须审批 | 尚未全额结清时下料由厂长审批 |
| 月结 | 客户按约定到期日结算 | 首次开通或修改月结条件时审批 | 正常未逾期不需要，逾期后需要 |
| 免单 | 当前订单不向客户收款 | 厂长审批免单原因和金额 | 免单审批通过后不再进行第二次审批 |

首期计划金额为 `0` 的“款未到先备料”归入分期付款，不建立新的付款方案类型。

现金、银行转账等是实际付款方式；建设银行、农业银行等是具体收款账户，都不是付款方案。

## 4. 最终应收与付款计划金额

- 付款计划以客户已经确认的当前报价版本最终应付金额为准。
- 基础报价默认不计税，报价总表可以填写本次实际税率，例如 `7%` 或 `8%`。
- 不计税时，最终应付为未税最终成交价。
- 计税时，最终应付为价税合计。
- 付款计划不自行重新计算报价、折扣或税额，只引用客户确认版本的最终应付金额。
- 分期付款各期计划金额之和必须等于当前最终应付金额。
- 免单同时保存原应收、免单金额、免单原因和实际应付 `0`。

报价、税率或成交价发生变化时，必须先形成新的报价和客户确认版本，再生成新的付款计划版本。

## 5. 付款计划建立与第一次厂长审批

### 5.1 客服建立计划

付款计划至少记录：

- 客户订单。
- 付款方案。
- 计划版本。
- 当前最终应收。
- 每期期次、计划金额和到期日期。
- 客服与客户确认的时间、方式和备注。
- 创建客服和创建时间。

付款计划允许多期，每期金额和日期必须明确。月结到期日由客服或财务根据实际约定填写，不由系统固定推算。

### 5.2 第一次厂长审批

- 分期付款计划必须由厂长审批后生效。
- 首期金额为 `0` 的计划同样必须审批。
- 免单必须由厂长审批免单原因和金额。
- 已获准且未逾期的月结客户，每张新订单不重复审批月结资格。
- 首次开通月结或修改月结条件时，由厂长审批。
- 全款计划不需要厂长审批。

第一次审批只表示厂长认可付款安排，不等于已经允许下料。

### 5.3 计划修改和版本

- 客服可以根据后续沟通修改尚未完成的金额或日期。
- 修改生成新的付款计划版本，不覆盖旧版本。
- 需要审批的付款方案在修改后重新提交厂长审批。
- 财务已经确认的到账记录保持不变。
- 系统使用新应收减去累计有效到账，重新计算剩余应收。

## 6. 付款提醒和逾期

分期和月结付款计划按每期到期日提醒：

- 到期前 3 天提醒客服和财务。
- 到期当天再次提醒客服和财务。
- 逾期后每天显示在逾期列表中，直至该期结清或计划被正式变更。

月结客户按客户整体控制：

- 任意旧客户订单到期未结清，该客户标记为“月结逾期”。
- 逾期前已经生产的货物锁定发货，等待厂长例外审批。
- 逾期后的新客户订单允许继续报价，但下料锁定，必须由厂长例外审批。
- 旧欠款到账并经财务确认后，系统自动解除客户级逾期限制。

## 7. 客户付款通知与财务到账确认

### 7.1 客服提交到账核对

客户通知已经付款后，客服提交到账核对申请，记录：

- 客户订单和关联付款期次。
- 客户声称付款金额。
- 付款方式。
- 付款人名称。
- 客户提供的备注、截图或凭证。
- 提交客服和提交时间。

客服提交只表示“客户声称已付款”，不计入累计到账。

### 7.2 财务核对

财务根据公司实际收款情况记录或核对：

- 实际到账金额。
- 到账日期和时间。
- 付款方式。
- 具体收款账户。
- 付款人名称。
- 银行流水号或备注。
- 收款凭证图片。
- 财务确认人和确认时间。
- 与客服提交金额存在差异时的差异说明。

现金允许没有银行流水号，但必须由财务确认实际收款。只有财务确认后的金额才计入累计到账。

### 7.3 确认、驳回和冲正

- 财务可以确认或驳回客服提交的到账核对申请。
- 一笔到账只能正式确认一次。
- 已确认记录不能直接覆盖金额、日期或账户。
- 发现录错、退款或银行退回时，通过冲正或作废记录纠正，并保留原记录。
- 免单没有到账记录。

## 8. 实际到账冲抵规则

实际到账金额不要求与某一期计划金额完全相同：

- 财务始终按真实到账金额确认。
- 系统优先冲抵最早到期且未结清的一期。
- 少付时，该期标记为“部分到账”，剩余金额继续待收。
- 多付时，先结清当前期次，超出部分继续冲抵下一期。
- 实际到账不会自动修改原付款计划。
- 客服与客户重新约定剩余金额或日期时，按付款计划新版本处理。

## 9. 付款方式与收款账户

### 9.1 付款方式

付款方式作为可维护基础资料，初始包括：

- 银行转账。
- 现金。
- 其他。

以后可以增加微信、支付宝等方式，不在业务代码中写死。

### 9.2 收款账户

收款账户独立维护：

- 账户名称。
- 开户银行。
- 账户类型：对公账户或其他授权账户。
- 账号信息。
- 是否启用。
- 排序和备注。

建设银行、农业银行、工商银行和交通银行等属于具体账户资料。停用账户保留历史到账快照，不允许从历史凭证中消失。

## 10. 第二次厂长下料放行审批

### 10.1 需要审批的情况

- 分期付款尚未全额结清但需要下料。
- 首期金额为 `0`，没有实际到账但需要下料。
- 月结客户已经逾期，需要例外下料。
- 其他未达到正常全款放行条件的例外情况。

### 10.2 不需要审批的情况

- 全款订单已经由财务确认全额到账。
- 正常且未逾期的有效月结客户。
- 免单已经完成厂长免单审批。

付款计划审批和下料放行审批是两个独立审批。付款计划已通过，不能代替下料放行审批。

### 10.3 厂长与老板沟通

需要第二次下料审批时：

```text
客服提交下料放行申请
  -> 厂长查看付款计划、应收和到账
  -> 厂长与老板沟通
  -> 厂长记录沟通结果并在系统审批
  -> 客服执行放行
```

- 客服和其他员工不直接联系老板申请放行。
- 老板不设置独立系统审批节点。
- 厂长记录沟通时间、老板意见和审批备注。
- 系统中的正式审批责任人是厂长。

## 11. 下料放行范围

- 下料放行申请绑定客户订单，不绑定单独工厂订单。
- 申请时保存客户订单当时全部“未放行且有效”的工厂订单快照，已经放行的工厂订单不重复进入。
- 厂长审批该客户订单本次快照范围是否允许下料。
- 客服一次性放行快照内全部未放行有效工厂订单，不选择其中部分订单放行。
- 已作废工厂订单不进入放行范围。
- 放行后新增的补单或新工厂订单不继承原放行结果，必须重新完成财务处理并建立新的放行周期。

## 12. 审批快照和自动失效

厂长下料审批必须保存：

- 当前报价和客户确认版本。
- 当前付款计划版本。
- 当前总应收、累计有效到账和剩余应收。
- 当时包含的有效工厂订单。
- 厂长与老板沟通记录。

出现以下情况，原审批自动失效，必须重新申请：

- 报价、税率、成交价或最终应收变化。
- 付款计划被修改。
- 财务撤销或冲正到账记录。
- 客户确认版本失效。
- 客户订单新增普通工厂订单。

客户补付导致累计有效到账增加时，不使原审批失效。一次审批只能执行一次客服放行，不能重复使用。

## 13. 客服执行放行

- 厂长审批通过后，客户订单状态变为“待客服放行”。
- 客服点击放行后，状态变为“已放行、待下料”。
- 放行记录保存客服、时间、审批记录和工厂订单快照。
- 本阶段只确认订单具备下料资格，不立即生成 BOM、不锁库存、不安排生产。
- 客服不能自行撤回已经执行的放行。

## 14. 厂长撤回放行

- 厂长在任何阶段都有权撤回下料放行，包括已经开始下料、已经锁定库存或已经生成生产任务之后。
- 撤回时必须填写明确原因，记录撤回人和撤回时间。
- 尚未开始下料时，订单停止进入下料队列，原放行失效。
- 已经开始下料或生产时，系统立即停止尚未执行的后续动作，并向下料、仓库和生产环节发出停止指令。
- 已完成的下料、库存变动、生产任务和现场记录不得删除或回滚成“未发生”，必须保留真实状态。
- 下游人员需要确认实际停止位置；具体停止、返库、报废或继续生产规则在后续生产变更流程中定义。
- 撤回后的订单如需恢复，必须重新提交放行审批并由客服重新执行放行。
- 放行、撤回、停止确认和重新放行全部永久留痕。

## 15. 主要状态流转

全款：

```text
待付款
  -> 客服提交到账核对
  -> 财务确认全额到账
  -> 待客服放行
  -> 已放行、待下料
```

分期付款：

```text
付款计划草稿
  -> 待厂长审批付款计划
  -> 付款计划已批准
  -> 客服提交到账核对
  -> 财务确认实际到账
  -> 待厂长审批下料放行
  -> 待客服放行
  -> 已放行、待下料
```

首期为 `0`：

```text
付款计划已批准
  -> 待厂长审批下料放行
  -> 待客服放行
  -> 已放行、待下料
```

正常月结：

```text
月结资格有效且未逾期
  -> 待客服放行
  -> 已放行、待下料
```

免单：

```text
免单待厂长审批
  -> 免单已批准
  -> 待客服放行
  -> 已放行、待下料
```

任意已放行订单均允许进入：

```text
已放行、待下料或下料中
  -> 厂长撤回
  -> 已撤回或暂停处理中
```

## 16. 权限边界

- 客服建立和修改付款计划、提交到账核对、提交放行申请并执行放行。
- 客服不能确认实际到账，不能代替厂长审批，也不能撤回已执行放行。
- 财务确认、驳回和冲正到账记录，不修改报价和厂长审批结果。
- 厂长审批付款计划、免单、月结条件和例外下料，并可以撤回放行。
- 需要老板意见时由厂长负责沟通，员工不直接向老板申请。
- 管理员处理系统异常，但不能代替客服、财务或厂长完成正常业务审批。

## 17. 流程结束状态

本流程正常结束时形成：

- 一个当前有效付款方案和付款计划版本。
- 全部有效到账、冲正和冲抵记录。
- 必要的付款计划审批和下料放行审批。
- 客服执行的整张客户订单放行记录。
- 客户订单状态“已放行、待下料”。

下一阶段从客户订单进入待下料队列开始，继续定义 BOM、库存锁定、采购、下料和排产。

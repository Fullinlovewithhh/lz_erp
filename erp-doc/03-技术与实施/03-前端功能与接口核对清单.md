# 前端功能与接口核对清单

> 文档性质：只读代码审计结果，不作为业务规则来源
>
> 扫描日期：2026-07-19
>
> 扫描范围：`erp-web/src`、前端 API 封装、后端控制器、角色权限和主要业务状态校验
>
> 本次没有修改数据库、后端或前端代码，也没有执行会写入业务数据的按钮测试

## 1. 状态说明

| 标记 | 含义 |
|---|---|
| 静态链路完整 | 按钮、处理函数、API 封装和后端接口均存在，静态字段基本对应；仍需浏览器和测试库验证 |
| 部分可用 | 接口存在，但角色、状态、字段或页面交互不完整 |
| 业务错配 | 按钮可以调用接口，但实现的是已经废止或尚未同步的新旧业务规则 |
| 假按钮 | 点击后只显示提示、没有业务接口，或显示在当前页面但对页面数据不产生作用 |
| 占位页面 | 菜单存在，但页面明确只显示“后续补齐” |

## 2. 总体结论

当前前端不能直接按“所有按钮都可用”理解。主要问题不是按钮没有绑定点击事件，而是页面结构、权限和业务版本没有同步。

最高优先级问题：

1. 非管理员登录初始化会调用当前角色无权访问的多个接口，客服、深化工程师、财务和厂长都可能在登录后立即出现接口拒绝或数据加载不完整。
2. `OrderFlowV3.vue` 将多个订单、报价、财务和配置菜单映射到同一综合页面，没有按角色和状态隐藏按钮；大量按钮“看得见但点击后被后端拒绝”。
3. 客户固定折扣前端字段没有写入正式 `default_discount_rate`；单张订单仍走已经废止的折扣申请和厂长审批。
4. L 与 V 应“报价分开、财务统一”，但当前 L 客户报价 PDF 会等待 V 报价确认，并把 V 外部报价金额写入 L PDF。
5. 付款计划后端有厂长审批接口和待办数据，但前端没有付款计划批准、驳回按钮，付款计划会停在“待厂长审批”。
6. 下料放行仍要求客服勾选“老板已口头同意”，并在放行时把客户订单下全部工厂订单改成待下料，没有保存本次未放行订单快照。
7. 客户数据库为空时，客户档案页自动显示六条演示客户；这些数据不是数据库真实数据，容易误判为系统仍有测试数据。

## 3. 公共界面与登录

| 页面或按钮 | 前端处理 | 后端或数据 | 结论 |
|---|---|---|---|
| 登录系统 | `doLogin` -> `/api/auth/login` | 接口存在，返回角色和模块 | 管理员静态链路完整；其他角色存在登录后初始化失败风险 |
| 退出登录 | `logout` -> `/api/auth/logout` | 接口存在 | 静态链路完整 |
| 左侧模块与子菜单 | 本地切换组件 | 模块权限来自登录返回值 | 部分可用；只控制模块，不控制子页面和按钮 |
| 收起侧栏、菜单搜索 | 前端本地状态 | 不需要接口 | 静态链路完整 |
| 顶部全局搜索 | `applyGlobalSearch` | 仍操作旧订单、旧报价页面状态 | 业务错配；对当前 V3 订单、报价和财务综合页基本无效 |
| 顶部筛选 | `openFilter` | 只有人员、产品、材料使用 `activeFilter` | 人员、产品、材料可用；账号、工艺、价格、生产线配置页面是假按钮 |
| 顶部导出 | `exportCurrent` | 仅实现人员、产品、材料 CSV | 人员、产品、材料可用；账号权限页仍显示导出但点击提示无数据 |

### 3.1 登录初始化错配

`App.vue` 登录成功后无条件加载产品、材料、人员、客户订单、旧报价、旧合同和价格规则。不同角色实际权限并不覆盖这些接口：

- 客服无权读取产品、材料和价格规则，但报价页面又需要这些数据。
- 深化工程师无权读取人员和客户列表，但初始化和工厂订单页面会读取。
- 财务无权读取客户、客户订单、产品、材料和价格规则，但综合财务页面初始化会读取。
- 厂长前端获得全部模块，API 层却基本只允许 `/api/order-flow`。
- 登录后仍停留在默认“基础资料/人员管理”，不会自动切换到当前角色第一个允许模块。
- 未识别角色如果返回空模块列表，前端会把空列表解释为“全部模块可见”，与后端拒绝访问的逻辑相反。

因此，很多按钮问题发生在点击之前：页面基础数据已经没有完整加载。

| 按钮 | 接口 | 结论 |

## 10. 财务数据

8.1 至 8.5 全部映射到同一个“报价结算与放行”页面，客服、财务、厂长看到的表单和待办没有按角色拆分。

### 10.1 8.1 付款计划

- “提交付款计划”接口存在，但前端每次只能建立一期，不能建立正式多期计划。
- 前端固定提交 `customerConfirmed=true`，没有客户确认记录输入。
- 后端将所有付款计划写成“待厂长审批”，没有按全款、分期、月结、免单区分审批规则。
- 后端提供厂长批准付款计划接口，工作队列也返回 `paymentPlans`。
- 前端没有渲染付款计划待办，也没有调用 `approvePaymentPlanV3` 的按钮。
- 结论：付款计划提交后可能永久停在待审批，属于缺失按钮。

### 10.2 8.2 到账确认

| 按钮 | 接口 | 结论 |
|---|---|---|
| 提交财务确认 | POST receipts | 接口存在，客服、财务或管理员可提交 |
| 财务确认/驳回 | POST receipt confirm | 财务或管理员可用；按钮向所有角色显示 |

字段缺失：

- 付款方式在页面写死，没有使用可维护基础资料。
- 前端没有到账日期输入，强制使用当前时间。
- 没有银行流水号或备注输入。
- 没有收款凭证图片上传。
- `paymentPlanId` 永远提交 `null`，到账无法明确绑定付款期次。
- 财务确认按钮没有要求填写财务备注。

### 10.3 8.3 价格调整审批

- 客服提交、厂长批准或驳回的接口存在。
- 申请和审批按钮都出现在同一综合页面，对非对应角色不隐藏。
- 历史审批列表没有独立展示，只显示当前待办。

### 10.4 8.4 下料放行

- 客服申请、财务确认、厂长审批、客服推送下料四组接口都存在。
- 当前仍由客服勾选“老板已口头同意”，与正式规则“厂长与老板沟通并记录”冲突。
- 没有审批快照明细表和本次未放行工厂订单列表。
- `releaseToCutting` 会把客户订单下全部工厂订单更新为“待下料”，包括已经放行或后来追加的订单。
- 没有厂长撤回、紧急停止、原因记录和重新放行按钮。
- 补单没有独立放行周期。

### 10.5 8.5 月结与逾期

- 页面只显示付款提醒列表。
- 没有月结客户维护、逾期解除、厂长例外批准、发货锁定和下料锁定操作界面。
- 状态：部分展示，不是完整月结管理。

### 10.6 8.6 收款账户配置

- 新增收款账户接口存在，财务或管理员可用。
- 页面同时显示公司、生产线和五金配置，财务能看见但无法操作其中多数按钮。
- 后端已有修改账户接口，前端没有编辑、启停或删除按钮。
- 厂长可以看到页面和新增按钮，但后端拒绝操作。

## 11. 权限核对

| 角色 | 前端现状 | 主要问题 |
|---|---|---|
| 管理员 | 全部模块可见 | 大多数静态链路可以加载，但仍受旧业务规则和占位页面影响 |
| 客服 | 客户、订单、报价模块 | 报价所需产品、材料和特殊规则接口无权限；综合页显示不属于客服的审批按钮 |
| 深化工程师 | 订单、报价模块 | 初始化读取人员和客户失败；可以通过过宽 API 创建或修改客户订单及基础资料 |
| 财务 | 订单、报价、财务模块 | 综合页初始化读取客户订单和主数据失败；看见客服和厂长按钮 |
| 厂长 | 前端全部模块 | API 只广泛允许 order-flow；大量基础资料按钮可见但点击失败 |
| 生产 | 订单、排单、生产模块 | 当前排单和生产均为占位；初始化会调用无权限接口 |

`/api/order-flow` 的网关权限允许客服、深化工程师、财务和厂长访问全部读取接口。虽然写接口内部有部分角色校验，但报价汇总、应收、到账和所有工作队列会被整个综合页面读取，读取权限仍然过宽。

## 12. 测试覆盖

当前只发现两个后端测试文件：

- `QuoteDetailServiceTest`：报价明细计算测试。
- `AuthServicePermissionTest`：部分 API 权限测试。

当前缺失：

- 前端组件测试。
- 浏览器按钮点击测试。
- 客服、深化工程师、财务、厂长的分角色端到端测试。
- 客户订单、CAD、拆单、报价、PDF、付款、到账和放行的完整流程测试。
- 按钮重复点击、刷新后状态和错误状态测试。

因此“静态链路完整”不等于已经经过真实运行验证。

## 13. 建议修复顺序

1. 修复登录初始化和模块首屏，让每个角色只加载自己有权读取的数据。
2. 将角色和状态权限落实到按钮显示、禁用原因和后端校验，不再让所有角色看到全部按钮。
3. 拆开 `OrderFlowV3` 的订单、报价、财务和配置视图，至少按模块显示对应功能。
4. 先修客户固定折扣和单张工厂订单折扣，再修 L/V 报价分离和税率输入。
5. 补齐付款计划厂长审批按钮、多期计划、到账凭证和期次绑定。
6. 按正式规则重做下料放行快照、厂长沟通记录、撤回和补单独立放行。
7. 删除客户档案演示数据回退和所有假按钮，或明确隐藏尚未实现的入口。
8. 每修完一个业务闭环，使用对应角色在浏览器真实点击，并保存测试结果后再提交 Git。

本清单只记录问题和接口关系，没有实施修复。

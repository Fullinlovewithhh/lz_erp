# ERP 数据库主结构设计

> 文档性质：正式数据结构说明
>
> 业务依据：`01-业务真源` 中已经确认的流程

## 1. 设计目标

本版本固定 ERP 核心订单结构，解决客户订单和工厂拆单混用的问题。

核心业务链路：

```text
客户 customer
  -> 客户订单 customer_order
    -> 工厂订单 factory_order（一张或多张）
      -> 报价、深化、排产、生产、财务
```

基本规则：

- 数据表之间使用主键建立关系，不使用名称建立关系。
- 客户和客户订单使用数据库内部数字 ID。
- 工厂订单使用英文数字订单号作为主键，不再重复设置数字 ID 和工厂订单号。
- 前端不显示客户 ID、客户订单 ID 等内部数字 ID。
- 一张客户订单可以拆成一张或多张工厂订单。
- 一张工厂订单只能属于一张客户订单。
- 客户原始 CAD 属于客户订单，深化图和生产图属于工厂订单。

## 2. 前后端字段口径

| 业务对象 | 后端主键 | 前端主要显示 | 说明 |
|---|---|---|---|
| 客户 | `customer.id` | `customer_name` 客户名称 | 数字 ID 只在后端使用 |
| 客户订单 | `customer_order.id` | `customer_order_no` | 客户提供的订单，订单号同时承担前端名称 |
| 工厂订单 | `factory_order.factory_order_id` | `factory_order_id`、`factory_order_name` | `factory_order_id` 同时是主键和工厂订单号 |

字段命名规则：

```text
id / xxx_id       主键或关联键
xxx_no            普通业务编号
xxx_code          基础资料编码
xxx_name          名称
xxx_snapshot      历史快照
```

特殊约定：`factory_order_id` 是英文数字组成的业务主键，中文统一显示为“工厂订单号”。

## 3. 客户表 customer

```text
customer
  id                 BIGINT PRIMARY KEY AUTO_INCREMENT
  customer_code      VARCHAR(50) NOT NULL
  customer_name      VARCHAR(100) NOT NULL
  phone              VARCHAR(30)
  address            VARCHAR(255)
  status             VARCHAR(30) NOT NULL
  created_at         DATETIME NOT NULL
  updated_at         DATETIME NOT NULL
```

约束：

- `customer_code` 全系统唯一。
- 客户名称用于前端选择和模糊搜索，不作为表之间的关联字段。
- 客户已有业务数据后只能停用，不允许物理删除。

## 4. 客户订单表 customer_order

客户订单表示客户传给工厂的一整张订单，是工厂拆单前的原始业务单元。

```text
customer_order
  id                    BIGINT PRIMARY KEY AUTO_INCREMENT
  customer_id           BIGINT NOT NULL
  customer_order_no     VARCHAR(255) NOT NULL
  customer_order_name   VARCHAR(255) NULL（仅兼容旧结构，新业务不使用）
  received_date         DATE NULL
  status                VARCHAR(30) NOT NULL
  remark                TEXT NULL
  is_deleted            TINYINT NOT NULL DEFAULT 0
  created_at            DATETIME NOT NULL
  updated_at            DATETIME NOT NULL
```

关系：

```text
customer_order.customer_id -> customer.id
```

客户订单号全系统不允许重复。后台仍使用 `customer_order.id` 建立关联，并对 `customer_order_no` 建立唯一约束；前端发现重复时直接阻止创建。

建议状态：

```text
待拆单 -> 拆单中 -> 已拆完 -> 已关闭
```

## 5. 工厂订单表 factory_order

工厂订单表示客户订单拆分后，工厂内部实际执行的订单。

```text
factory_order
  factory_order_id       VARCHAR(50) PRIMARY KEY
  customer_order_id      BIGINT NOT NULL
  factory_order_name     VARCHAR(255) NOT NULL
  split_sequence         INT NOT NULL
  engineer_id            BIGINT NULL
  status                 VARCHAR(50) NOT NULL
  demand_desc            TEXT NULL
  created_by             BIGINT NULL
  created_at             DATETIME NOT NULL
  updated_at             DATETIME NOT NULL
```

关系：

```text
factory_order.customer_order_id -> customer_order.id
```

规则：

- `factory_order_id` 同时是数据库主键和前端显示的工厂订单号。
- `factory_order_id` 只能包含英文字母和数字，建议统一大写。
- `factory_order_id` 创建后永久禁止修改。
- 不再设置 `id`、`factory_order_no`、`contract_no` 三套重复编号。
- `factory_order_name` 是拆单名称，例如“住宅1-客厅”。
- `split_sequence` 表示同一客户订单下的拆单顺序。
- 一个工厂订单只能绑定一个 `customer_order_id`。

建议检查约束：

```text
factory_order_id REGEXP '^[A-Z0-9]+$'
split_sequence > 0
UNIQUE (customer_order_id, split_sequence)
```

## 6. 业务示例

客户“客户1”传来住宅1订单：

```text
customer
  id = 1
  customer_name = 客户1

customer_order
  id = 101
  customer_id = 1
  customer_order_no = KH20260001

factory_order
  factory_order_id = GC2026000101
  customer_order_id = 101
  factory_order_name = 住宅1-客厅
  split_sequence = 1

factory_order
  factory_order_id = GC2026000102
  customer_order_id = 101
  factory_order_name = 住宅1-卧室
  split_sequence = 2
```

前端显示：

```text
客户1
└── KH20260001 / 住宅1
    ├── GC2026000101 / 住宅1-客厅
    └── GC2026000102 / 住宅1-卧室
```


## 8. 下游业务关联

生产线需要增加报价模式字段：

```text
production_line.quote_mode VARCHAR(20) NOT NULL
```

取值为 `ERP`、`EXTERNAL` 或 `NONE`。当前实木生产线使用ERP内部报价，板式生产线使用外部系统报价。

### 8.1 正式报价结构

完整报价编辑器统一绑定 `factory_order.factory_order_id`，不再写入旧 `contract_id` 报价链。

```text
factory_order
  -> factory_order_quote（报价版本，一张工厂订单一个当前有效版本）
    -> factory_order_quote_item（完整产品、尺寸、材质、工艺和图片明细）
      -> factory_order_quote_item_extra_price（规则及自定义工艺加价）
```

- 草稿反复保存使用同一版本；正式提交后才成为工厂订单当前报价版本。
- L 新版本正式提交后，原 L 客户确认和原 L 报价 PDF 自动失效。
- 产品和特殊工艺加价合并计算原价；每条特殊加价可单独设置是否参与折扣。
- 五金配件加入报价总额但始终不参与折扣。
- 工厂订单折后价汇总到客户订单，成交价调整必须经厂长审批。
- 税额在未税报价合计之后计算。
- L 客户报价 PDF 按版本保存，包含工厂订单号、完整明细、小图、折扣、五金、税额及价税合计。

### 8.2 报价主表 factory_order_quote

`factory_order_quote` 表示一张工厂订单的一个完整报价版本。它是报价主表，不直接保存客户订单下其他工厂订单的汇总明细。

核心字段：

```text
id                  BIGINT PRIMARY KEY AUTO_INCREMENT
factory_order_id    VARCHAR(50) NOT NULL
version_no          INT NOT NULL
status              VARCHAR(30) NOT NULL
original_amount     DECIMAL(14,2) NOT NULL
discount_rate       DECIMAL(6,4) NOT NULL
discount_amount     DECIMAL(14,2) NOT NULL
final_amount        DECIMAL(14,2) NOT NULL
quote_desc          VARCHAR(500) NULL
created_by          BIGINT NOT NULL
created_at          DATETIME NOT NULL
submitted_at        DATETIME NULL
```

正式关系：

```text
factory_order_quote.factory_order_id
  -> factory_order.factory_order_id
```

约束：

```text
UNIQUE (factory_order_id, version_no)
```

业务规则：

- 一张工厂订单可以有多个报价历史版本，但只能有一个当前有效版本。
- 草稿阶段反复保存同一个版本，不因每次保存产生新版本。
- 正式提交或已提交报价再次修改时，生成新的 `version_no`。
- 工厂订单通过 `current_quote_version` 指向当前有效版本。
- 历史版本只读，不能覆盖、删除或改写金额。
- 报价负责人转交时，当前报价草稿和明细继续保留在工厂订单下。
- L 新报价版本提交后，与旧版本关联的 L 客户确认和报价 PDF 自动失效。

### 8.3 报价明细表 factory_order_quote_item

`factory_order_quote_item` 保存一个报价版本中的产品、五金及其数量、尺寸、材料、工艺和金额，是报价计算的事实明细。

正式关系：

```text
factory_order_quote_item.quote_id
  -> factory_order_quote.id
```

主要字段分组：

| 字段组 | 主要字段 | 用途 |
|---|---|---|
| 分类与来源 | `product_category`、`product_id`、`hardware_item_id` | 区分产品和五金，并关联基础资料 |
| 名称与规格 | `product_name`、`specification`、`material_structure`、`handle_color` | 保存报价时的业务快照 |
| 尺寸 | `width_mm`、`height_mm`、`thickness_mm`、`area_m2` | 记录尺寸和参与计价的面积 |
| 工艺与附件 | `hinge_hole`、`process_desc`、`production_process`、`attachment_path` | 保存报价和后续生产需要的信息 |
| 数量与单位 | `quantity`、`unit` | 支持平方米、米、个、套等计价单位 |
| 单价 | `base_unit_price`、`special_adjust_total`、`final_unit_price`、`unit_price` | 保存基础价、工艺加价和最终单价 |
| 金额 | `original_amount`、`discount_amount`、`final_amount` | 保存原价、折扣额和最终金额 |
| 折扣 | `discount_eligible` | 控制该明细是否参与工厂订单折扣 |
| 排序与备注 | `sort_order`、`remark` | 控制PDF顺序并保存补充说明 |

业务规则：

- 报价明细必须属于一个明确的报价版本。
- 产品、尺寸、材料和工艺等内容在报价版本中保存快照，基础资料以后修改不能改变历史报价。
- 五金通过 `product_category='HARDWARE'` 标识，并可以关联 `hardware_item_id`。
- 五金进入报价总额，但 `discount_eligible` 默认且固定为不参与折扣。
- 其他产品默认参与折扣，允许报价人员人工取消“参与折扣”。
- 明细金额必须由后端报价引擎计算，前端不能直接提交一个无法复算的总金额。
- 报价明细已有正式版本后不得物理删除；修改时生成新报价版本。

### 8.4 特殊加价表 factory_order_quote_item_extra_price

`factory_order_quote_item_extra_price` 保存某条报价明细采用的每一项特殊工艺加价，包括从规则库选择的规则和报价人员临时建立的自定义规则。

正式关系：

```text
factory_order_quote_item_extra_price.quote_item_id
  -> factory_order_quote_item.id

factory_order_quote_item_extra_price.quote_id
  -> factory_order_quote.id
```

核心字段：

```text
id                  BIGINT PRIMARY KEY AUTO_INCREMENT
quote_item_id       BIGINT NOT NULL
quote_id            BIGINT NOT NULL
source_rule_id      BIGINT NULL
rule_name           VARCHAR(100) NOT NULL
adjust_mode         VARCHAR(50) NOT NULL
adjust_value        DECIMAL(14,4) NOT NULL
unit_desc           VARCHAR(50) NULL
rule_quantity       DECIMAL(12,4) NULL
final_charge        DECIMAL(14,2) NOT NULL
discount_eligible   TINYINT NOT NULL DEFAULT 1
created_at          DATETIME NOT NULL
```

业务规则：

- 每一项特殊加价必须绑定具体报价明细和报价版本。
- `source_rule_id` 只记录规则来源；`rule_name`、计价方式、计价值和最终加价必须保存版本快照。
- 规则库后续修改或停用不能改变历史报价中的特殊加价。
- 自定义加价的 `source_rule_id` 可以为空，但规则名称、计价方式、数值和最终金额必须完整保存。
- 特殊加价默认参与折扣，但每一条可以单独设置为不参与折扣。
- 不参与折扣的特殊工艺仍计入最终应收，只是不乘工厂订单折扣率。
- 特殊加价记录随报价版本保留，不得跨版本复用同一行数据。

### 8.5 报价计算顺序

单条普通产品报价按以下顺序计算：

```text
基础产品金额
+ 参与折扣的特殊工艺加价
= 参与折扣原价

参与折扣原价 × 工厂订单折扣率
+ 不参与折扣的产品或特殊工艺
= 产品及工艺折后金额
```

五金单独计算：

```text
五金数量 × 五金报价单价
= 五金金额
```

五金金额加入报价总额，但不乘工厂订单折扣率。

客户订单汇总顺序：

```text
全部工厂订单产品及工艺折后金额
+ 全部五金金额
+ 客户订单成交价调整
= 未税报价合计

未税报价合计 × 税率
= 税额

未税报价合计 + 税额
= 价税合计
```

税率保存在客户订单报价总表或客户报价 PDF 版本中，而不是写死在产品或工厂订单报价明细中。前端按百分比输入，例如 `7`、`8`，后端分别按 `0.07`、`0.08` 参与计算。基础报价默认税率为 `0`。

税率必须在生成客户报价 PDF 前确定。当前 PDF 的未税合计、税率、税额和价税合计共同构成客户确认快照；修改税率必须生成新的 PDF 版本。付款计划引用客户确认版本的最终应付金额，不自行重新计算税额。

成交价调整允许为负数，用于客户要求抹零或最终议价，但必须生成价格调整版本并经过厂长审批。已经确认的到账记录不因报价修改而删除，系统重新计算剩余应收。

### 8.6 报价数据保留原则

- 报价主表、报价明细和特殊加价共同组成一个不可拆分的报价版本。
- 正式提交后不允许只修改其中一张表而不生成新版本。
- 金额字段使用 `DECIMAL`，禁止使用浮点类型保存业务金额。
- 金额计算和四舍五入规则统一放在后端报价引擎。
- 数据库保存计算结果，后端必须能够根据明细和规则快照重新复算并校验结果。
- 报价记录原则上不物理删除，只能撤回、失效或生成新版本。

### 8.7 产品动态计价结构

产品类别、产品、计价单位和计价模式必须来自基础资料，不能写死在报价页面。

```text
product_category
  -> product
    -> factory_order_quote_item（保存报价快照）
```

每个具体产品只允许一种计价模式：面积、长度或数量。产品资料保存基础单价、单位、需要填写的尺寸字段和可选最低计价规则。报价明细保存实际尺寸、系统计价数量、基础单价和金额快照。

### 8.8 特殊计价规则结构

常用特殊计价规则来自基础资料；不常用费用只保存在当前报价明细中，不自动进入规则库。

```text
special_pricing_rule_category
  -> special_pricing_rule

factory_order_quote_item
  -> factory_order_quote_item_extra_price
```

特殊加价在产品基础价格之上增加，支持按面积、长度、数量或固定一次性计价。所有特殊加价绑定报价明细，报价主表只做汇总展示。规则库变化不能改变历史报价快照。

### 8.9 V外部报价结构

V板式生产线不建立ERP产品报价明细，每张V工厂订单保存一条当前外部报价结果及历史修改记录：

```text
factory_order_external_quote
  id
  factory_order_id
  external_quote_no
  version_no
  final_amount
  quote_date
  attachment_path
  confirmation_status
  confirmed_at
  recorded_by
  recorded_at
```

V外部报价金额参与客户订单财务应收，但不进入L实木报价PDF。

### 8.10 客户折扣结构

客户固定折扣与某张工厂订单的临时折扣分开保存：

```text
customer_discount_policy
  -> 客户长期固定折扣、有效期和批准记录

factory_order_discount_request
  -> 单张L工厂订单临时折扣申请和审批记录
```

新客户默认十折。固定折扣默认应用到该客户所有L工厂订单；降低折扣需要销售经理或老板审批。折扣变化必须生成新报价版本。

### 8.11 报价完成与统一应收

```text
客户订单总应收
= 已确认L实木报价
+ 已确认V板式外部报价
```

L全部工厂订单整单确认、V全部外部报价分别确认、折扣及成交价调整全部审批完成后，客户订单才能进入付款计划。已经确认到账的财务记录不因报价版本变化而删除。

工厂订单级报价和任务使用 `factory_order_id`，付款、财务应收和下料放行使用 `customer_order_id`。当前正式关系：

```text
factory_order_assignment_log.factory_order_id -> factory_order.factory_order_id
factory_order_discount_request.factory_order_id -> factory_order.factory_order_id
factory_order_quote.factory_order_id -> factory_order.factory_order_id
factory_order_external_quote.factory_order_id -> factory_order.factory_order_id

customer_order_price_adjustment.customer_order_id -> customer_order.id
customer_quote_pdf.customer_order_id -> customer_order.id
customer_quote_confirmation.customer_order_id -> customer_order.id
customer_payment_plan.customer_order_id -> customer_order.id
customer_payment_receipt.customer_order_id -> customer_order.id
cutting_release_request.customer_order_id -> customer_order.id
```

下游表可以保存工厂订单名称、客户订单号或客户名称快照用于历史展示，但快照不能代替正式关联字段。

## 9. 删除和修改规则

- 客户和客户订单使用停用或软删除。
- 已产生工厂订单的客户订单不能物理删除。
- 已被下游表引用的工厂订单不能物理删除。
- `factory_order_id` 创建后不可修改；录错时应作废并重新创建。
- 客户订单号如需修改，应记录操作日志。
- 禁止通过客户名称、客户订单号或工厂订单名称直接关联业务表。
